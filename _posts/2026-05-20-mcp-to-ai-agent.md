## 一、从工具到智能：一个关键转折

上两篇文章分别讨论了工具调用（Function Calling）和 MCP（Model Context Protocol）集成。工具调用让模型能"做事"，MCP 让工具接入有了统一标准。这两件事加在一起，指向一个更大的话题——**AI Agent**。

这不是偶然。2024 年以来，"Agent" 成了 AI 领域最高频的词汇之一。Anthropic、OpenAI、Google 都在推自己的 Agent 框架。但如果你已经理解了 MCP，其实离理解 Agent 只差一层窗户纸。

本文的目标就是捅破这层窗户纸。

---

## 二、回顾 MCP：给大模型配一个标准工具箱

先快速回顾 MCP 解决了什么问题。

在没有 MCP 之前，每接一个外部工具都要写定制代码：API 格式不一样、认证方式不一样、错误处理不一样。你想让 AI 查天气，写一套；想让 AI 读文档，再写一套。工具越多，适配代码越膨胀。

MCP 做的事很简单：**定义了一个标准协议**。

```
┌──────────────┐     JSON-RPC 2.0      ┌──────────────┐
│              │ ◄──────────────────► │              │
│  MCP Client  │    tools/list         │  MCP Server  │
│  (AI 应用)    │    tools/call         │  (工具提供方)  │
│              │    initialize         │              │
└──────────────┘                       └──────────────┘
```

关键设计就三个动作：

1. **`initialize`** — 握手，交换能力信息
2. **`tools/list`** — 服务端告诉客户端"我会什么"
3. **`tools/call`** — 客户端让服务端"做某件事"

有了这个协议，工具提供方只需实现一个 MCP Server，任何 MCP Client 都能直接调用。工具消费者也不需要关心每个工具的细节——统一通过 `tools/list` 发现能力，通过 `tools/call` 执行操作。

这就像 USB 接口：键盘、鼠标、U 盘各不相同，但插上都能用。

但 MCP 本身只是"管道"。它解决了**怎么接**的问题，没有解决**接上之后做什么**的问题。后者，就是 Agent 的范畴。

## 三、什么是 AI Agent？

Agent 这个词被用得很泛，但它的核心定义其实很明确。一个 AI Agent 必须具备四个能力：

### 3.1 感知（Perception）

Agent 需要感知环境。环境可以是用户输入的文字、对话上下文、外部数据（天气、股价、文件内容），也可以是更复杂的多模态信息（图片、音频）。

MCP 在这里扮演"眼"的角色——通过 `tools/list` 和 `tools/call`，Agent 获得了感知外部世界的能力。

### 3.2 规划（Planning）

这是 Agent 区别于普通 chatbot 最核心的能力。普通 chatbot 是"一问一答"：你问，它答。Agent 是"给目标，它拆解"：你说"帮我安排明天下午的会议"，Agent 会自己拆成：

1. 查日历看明天下午有没有冲突
2. 查参会人的空闲时间
3. 预定会议室
4. 发会议邀请邮件

每一步都可能失败，Agent 需要根据中间结果调整后续步骤。这就是规划。

### 3.3 记忆（Memory）

记忆分两种：

- **短期记忆**：当前的对话上下文，知道刚才发生了什么
- **长期记忆**：跨会话的信息，比如用户偏好、历史决策、知识积累

没有记忆的 Agent 就像每次见面都重新认识你的同事——能干活，但效率极低。

### 3.4 行动（Action）

有了感知、规划和记忆，最后一步是执行。行动可以是调用 API、操作文件、发送消息、执行代码——任何改变外部世界状态的操作。

MCP 在这里扮演"手"的角色——通过标准化的工具调用，Agent 将规划转化为实际行动。

### 一个完整的 Agent 循环

```
用户输入 → 感知环境 → 分析意图 → 制定计划 → 选择工具 → 执行动作
                                                    ↑       ↓
                                                    └─ 观察结果 ←─┘
                              (循环直到目标达成)
```

这个循环被称为 **ReAct** (Reasoning + Acting) 模式，是目前大多数 Agent 框架的核心范式。

## 四、MCP 如何成为 Agent 的"手"和"眼"

让我们用一个具体例子来串联 MCP 和 Agent 的工作方式。

### 场景：用户说"帮我看看这周 GitHub 上 Java 项目有什么新动态"

**第一步：感知 — Agent 理解意图**

Agent 从用户输入中提取关键信息：
- 目标：了解 Java 项目动态
- 范围：本周
- 数据源：GitHub

**第二步：规划 — Agent 拆解任务**

Agent 的推理过程（内部思维）：

> 1. 需要搜索 GitHub 上本周创建的 Java 仓库 → 调用 `github_search_repos`
> 2. 可能需要了解热门项目的详细信息 → 可能需要额外工具
> 3. 最后要组织成一篇可读的摘要 → 需要文本生成能力

**第三步：行动 — MCP 工具调用**

Agent 通过 MCP Client 向 GitHub MCP Server 发起调用：

```json
// 请求 (JSON-RPC 2.0)
{
  "jsonrpc": "2.0",
  "method": "tools/call",
  "params": {
    "name": "github_search_repos",
    "arguments": {
      "query": "Java",
      "language": "Java",
      "sort": "updated",
      "per_page": 10
    }
  }
}

// 响应
{
  "jsonrpc": "2.0",
  "result": {
    "content": [
      {
        "type": "text",
        "text": "找到 10 个仓库：\n1. spring-projects/spring-ai ...\n..."
      }
    ]
  }
}
```

**第四步：观察与迭代**

Agent 拿到结果后评估：
- 信息够吗？→ 如果不够，调整搜索参数再查
- 需要深入某个仓库吗？→ 调用 `github_search_code` 查看代码
- 结果整理好了吗？→ 组织成用户友好的格式输出

**在整个过程中，MCP 做了两件事：**
- **作为"眼"**：通过 `tools/list` 发现 GitHub MCP Server 提供了搜索仓库和搜索代码两个能力
- **作为"手"**：通过 `tools/call` 将 Agent 的规划转化为实际的 GitHub API 调用，并把结果带回来

MCP 不参与规划，不参与记忆管理，不参与结果评估。它就是纯粹的"感官-动作"层——这正是它设计上的优雅之处：**职责单一，边界清晰**。

## 五、对比：普通 Chatbot vs AI Agent

这个对比不是"谁更好"，而是"谁适合什么场景"。

| 维度 | 普通 Chatbot | AI Agent |
|------|-------------|----------|
| **交互模式** | 一问一答，无状态 | 多轮循环，有状态 |
| **能力边界** | 仅限训练数据中的知识 | 可调用外部工具获取实时信息 |
| **任务类型** | 回答、解释、总结 | 搜索、分析、执行、验证 |
| **自主性** | 被动响应 | 主动规划、执行、纠错 |
| **记忆** | 仅当前会话 | 短期+长期记忆 |
| **错误处理** | 重新提问 | 自动重试、换策略 |
| **典型场景** | "解释一下 JVM 内存模型" | "帮我找 3 个开源的 MCP 实现，比较它们的优劣，整理成表格" |

用一个具体的例子来感受区别：

**用户**："明天下午 3 点开会方便吗？"

**普通 Chatbot 的回答**：
> 我无法访问你的日历，建议你打开日历应用检查一下明天下午 3 点是否有空闲。

**AI Agent 的行为**：
1. 调用日历工具，查询明天下午 3 点的安排
2. 发现 3 点已有一个周会
3. 查询 3 点之后最近的空闲时段：4 点
4. 回复用户："明天下午 3 点有周会（预计 1 小时），4 点之后有空。需要我帮你把新会议定在 4 点吗？"

区别不在于回答质量，而在于 **Agent 能主动获取回答所需的信息**。

## 六、从 MCP 到 Agent 的思维跃迁

如果你理解了下面的递进关系，就抓住了本质：

```
Function Calling  →  单次工具调用
        ↓
MCP              →  标准化工具接入协议
        ↓
Tool Ecosystem   →  多样化的工具供给
        ↓
Agent            →  自主规划 + 工具编排 + 闭环执行
```

**MCP 是 Agent 的基础设施**。没有标准化的工具接入，每个 Agent 都要重复造轮子。有了 MCP，Agent 开发者可以假设"世界上的所有工具都遵循同一个协议"，然后专注于更上层的问题：如何规划、如何记忆、如何纠错。

这也是为什么 Anthropic 在推出 Claude Agent 的同时大力推 MCP——它们不是两个独立的产品，而是一个生态的两层：

- **下层 MCP**：连接模型与外部世界
- **上层 Agent**：赋予模型自主行动的能力

### 当前 Agent 的局限性

坦率地说，今天的 Agent 还远不够成熟：

- **规划不可靠**：复杂任务的拆解经常出错，中间步骤偏离目标
- **记忆很原始**：长期记忆大多还是"把东西存进向量数据库然后检索"，离真正的记忆差很远
- **工具选择的鲁棒性不足**：工具多了之后，Agent 可能选错工具或用错参数
- **成本高**：一个复杂任务可能触发几十次 LLM 调用，延迟和费用都不低

但这些不是 MCP 的问题——MCP 已经把自己那层做好了。剩下的是上层 Agent 框架要解决的。

---

## 七、总结

这篇文章的核心论点可以归纳为三句话：

1. **MCP 解决了工具的标准化接入问题**——它是 Agent 的"感官-动作层"
2. **Agent = 感知 + 规划 + 记忆 + 行动**——MCP 覆盖了感知和行动，规划和记忆是 Agent 框架的职责
3. **从 MCP 到 Agent 是一次思维跃迁**——从"模型调用工具"到"模型自主使用工具完成任务"

如果你正在做 MCP 相关的开发，下一步自然会走向 Agent。而且好消息是，你不需要从零开始——MCP 已经帮你打好了最底层的地基。

引用一句来自 Anthropic 的总结：

> "MCP is to AI agents what USB-C is to devices — a universal standard that lets anything connect to anything."

Agent 的时代才刚刚开始。
